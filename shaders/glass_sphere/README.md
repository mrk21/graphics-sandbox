# Glass Sphere シェーダー — ブラックホール描画

## 概要

OpenGL フラグメントシェーダーによるリアルタイムレイトレーシングシーン。ガラス球、金属ブロック、草ブロック、チェッカー床、スポットライトに加え、**シュワルツシルト計量に基づくブラックホールの重力レンズ効果**を実装。

## ブラックホール描画のロジック

### 物理モデル

シュワルツシルト・ブラックホール（非回転・非帯電）をベースにしている。

| パラメータ              | 値     | 説明                                               |
| ----------------------- | ------ | -------------------------------------------------- |
| シュワルツシルト半径 rs | 0.24   | 事象の地平面 = rs                                  |
| 光子球                  | 1.5 rs | 光が周回軌道を描く半径                             |
| 光子捕獲半径            | 2.6 rs | これより内側の光は吸収される（BHの「影」のサイズ） |
| 影響半径                | 2.5    | 重力レンズの計算が行われる範囲                     |

### レイマーチング

カメラからのレイがBH影響圏に入ると、直線的なレイトレースではなくレイマーチングに切り替わる。

1. **影響圏入口までジャンプ**: レイと影響球の交差を計算し、入口までスキップ
2. **重力偏向の逐次計算**: 各ステップで一般相対性理論準拠の加速度を適用してレイの方向を曲げる
3. **適応的ステップサイズ**: BH近傍ではステップを極小に（`distRatio² × 0.3`、最小0.05）

### 重力の計算式

```glsl
// 一般相対性理論偏向: ニュートン予測の2倍（一般相対論の予測値）
float grFactor = 1.5 * rs / (dist * dist);

// 光子球付近でのブースト: 光がほぼ周回する効果を再現
float photonBoost = 1.0 + 2.0 * exp(-(dist - photonSphere)² / (rs² × 0.5));

// 影響圏境界でのスムーズなフェードアウト
float edgeFade = smoothstep(influenceR, influenceR * 0.5, dist);
```

ニュートン力学では光の偏向角は `2GM/(c²b)` だが、一般相対性理論ではその2倍の `4GM/(c²b)` になる（1919年の日食観測で確認）。この実装では `1.5 * rs / r²` という形で一般相対性理論補正を組み込んでいる。

### 吸収判定

事象の地平面 (rs) ではなく**光子捕獲半径 (2.6 rs)** で吸収判定を行う。これはシュワルツシルト計量において、この半径より内側を通過する光子はすべて事象の地平面に落ちるため。視覚的にもBHの「影」はこのサイズになる。

## 解決した技術的課題

### 1. レイが影響圏に入れない問題

**症状**: BHが全く見えない

**原因**: カメラが影響圏の外にいる場合、レイマーチの1ステップ目で「影響圏外」と判定されて即終了していた

**解決**: レイと影響球の交差を計算し、入口までレイをジャンプさせてからマーチを開始

### 2. シーンオブジェクトが切り取られる問題

**症状**: ガラス球がBH影響圏の端で不自然に切断される

**原因**: BH影響圏がシーンオブジェクトより近い場合、レイを影響圏全体で曲げてから再トレースするため、途中にあるオブジェクトを飛ばしていた

**解決**: レイマーチ中に4ステップごとにシーン交差をチェック。影響圏内でオブジェクトに近づいたら、曲がったレイの状態でそこから通常のトレースに戻す

### 3. ボリューメトリックライトの前後関係

**症状**: BH背後の光の筋がBH前面に表示される

**原因**: ボリューメトリックライトがポストエフェクトとして直線レイで計算されており、BHのレンズ処理と独立していた

**解決**: ボリューメトリックライトを `trace()` 関数に統合し、レイの各セグメントに沿って蓄積する方式に変更:
- カメラ → BH影響圏入口: 直線セグメント
- BH影響圏内: 曲がったレイに沿って蓄積（レイマーチ中に計算）
- BH影響圏出口 → シーンヒット: 偏向後の直線セグメント

### 4. 影響圏境界のくっきりした線

**症状**: BH周囲に不自然な円形の境界線が見える

**原因**: 重力計算が影響半径でハードカットオフしていた + ボリューメトリックライトも影響圏で急にカットしていた

**解決**:
- 重力を `smoothstep` で影響圏の外側50%〜100%の範囲でフェードアウト
- ボリューメトリックライトはカメラレイの最接近距離で自然に減衰

### 5. BH下部の縞模様アーティファクト

**症状**: BHの下部に縞模様の影と星が表示される

**原因**: 事象の地平面ギリギリを通過するレイの数値精度不足。隣接ピクセルのレイが全く違う方向に飛んでしまう（カオス的振る舞い）+ レイマーチ中のシーン交差チェックが粗い（8ステップごと）

**解決**:
- 吸収判定を光子捕獲半径 (2.6 rs) に拡大
- 適応的ステップサイズを二次関数的に縮小（BH近傍で最小0.05）
- シーン交差チェック頻度を4ステップごとに増加
- マーチステップ数を128に増加

## RTコア（ハードウェアレイトレーシング）との比較

### RTコアとは

Vulkan や DirectX 12 の RTパイプラインで利用できるハードウェアアクセラレータ（NVIDIA RTX の RT コア等）。本質的には「**直線レイと BVH（境界ボリューム階層）内の三角形ポリゴンとの交差判定**」を超高速に解く専用回路。

### 高速化できる部分

| 処理                         | 現在の実装 (OpenGL フラグメントシェーダー) | RTコア使用時                   |
| ---------------------------- | ------------------------------------------ | ------------------------------ |
| レイ vs オブジェクト交差判定 | ループで全オブジェクト総当たり             | ハードウェア BVH で O(log n)   |
| 反射・屈折の再帰レイ         | 手動ループ展開、回数制限あり               | 専用パイプラインで自然に再帰   |
| 複雑なメッシュ（多ポリゴン） | SDF 近似が必要                             | 実ポリゴンメッシュを直接扱える |

オブジェクト数が数百〜数千になると桁違いの差が出る。ただし現在のシーン規模（球・箱・平面の数個）では、交差判定が解析的に解けるため体感差は限定的。

### 高速化できない部分 — ブラックホールの重力レンズ

**RTコアでは原理的に重力レンズ効果を高速化できない。** 理由は以下の通り:

1. **ポリゴンが存在しない** — RTコアは「レイ vs 三角形」の交差判定ハードウェアであり、当たる対象（幾何学的な面）がなければ出番がない
2. **レイの経路が直線ではない** — 重力レンズは時空の歪みによりレイが連続的に曲がる。RTコアは直線レイしか扱えない
3. **屈折とは本質的に異なる** — ガラスの屈折は媒質の境界面でスネルの法則を離散的に適用（入射・出射の2回）するだけだが、重力レンズは空間の各点で微小に方向が変わり続ける（微分方程式の数値積分）

|                  | ガラス球の屈折           | ブラックホールの重力レンズ |
| ---------------- | ------------------------ | -------------------------- |
| 境界面           | ある（球面）             | ない                       |
| 方向が変わる回数 | 入射時・出射時の2回      | 連続的に無限回             |
| 計算手法         | スネルの法則を2回適用    | 測地線方程式の数値積分     |
| RTコア           | 球面との交差判定に使える | **使えない**               |

### 業界での実装例

重力レンズを実装している作品はすべてシェーダーによる手計算を採用しており、RTコアは使用していない。

| 作品・エンジン      | 手法                                                             |
| ------------------- | ---------------------------------------------------------------- |
| Interstellar (映画) | CPU でシュワルツシルト測地線を数値積分（オフラインレンダリング） |
| Elite Dangerous     | フラグメントシェーダーでポストエフェクト的に歪ませる             |
| Space Engine        | シェーダーでレイマーチ                                           |
| Unreal Engine 5     | 通常オブジェクトは RTコア、重力レンズ部分はカスタムシェーダー    |

### ハイブリッド方式の可能性

RTコアとレイマーチの組み合わせは理論的に可能:
- **BH 影響圏外**: RTコアで通常のレイトレーシング（反射・屈折・影）
- **BH 影響圏内**: レイマーチで重力偏向を計算
- **BH 影響圏を抜けた後**: 曲がったレイの出口から再び RTコアで直線レイの交差判定

本実装は OpenGL フラグメントシェーダーのみで完結するアプローチを採用しており、RTX GPU を必要とせず幅広い環境で動作する。

## ファイル構成

```
shaders/glass_sphere/
├── main.cpp                    # OpenGL/GLFW アプリケーション
├── CMakeLists.txt              # ビルド設定（MinGWクロスコンパイル）
├── shaders/
│   ├── vertex.glsl             # 頂点シェーダー（フルスクリーンクアッド）
│   └── fragment.glsl           # フラグメントシェーダー（レイトレーシング本体）
└── gen/                        # ビルド出力ディレクトリ
```

## ビルド・実行

```bash
cd shaders/glass_sphere
cmake -B gen
cmake --build gen
cd gen && ./glass_sphere.exe
```
