## 水中表現のロジック

### 水中で起きる2つの現象

1. **吸収（ビア・ランベルトの法則）** — 水が赤い光を吸収し、遠くが青くなる
2. **散乱（インスキャッタリング）** — 水自体が青い光を目に向けて散乱し、霧のように霞む

```glsl
vec3 absorption = vec3(0.2, 0.04, 0.03); // 赤を強く吸収、青はほぼ通す
vec3 scatterColor = vec3(0.05, 0.12, 0.15);

vec3 transmittance = exp(-absorption * dist);           // 吸収: 遠いほど赤が消える
vec3 inScatter = scatterColor * (1.0 - transmittance);  // 散乱: 遠いほど青が足される

finalColor = hitColor * transmittance + inScatter;
```

### ボリューメトリックライトとの関係

- ボリューメトリックライト: レイに沿って光の寄与を**蓄積**する
- 水中の減衰: レイに沿って光が**失われていく**

方向は逆だが「レイに沿って媒質の効果を積分する」という枠組みは同じ。両方を同じループ内で処理できる。

### ペライチ水面 vs 水ブロック（Minecraft 方式）

|                | ペライチ水面            | 水ブロック                           |
| -------------- | ----------------------- | ------------------------------------ |
| 形状           | 無限平面                | 任意の形（ブロック単位で配置）       |
| 水中距離の計算 | 入射点〜ヒット点の1区間 | 複数の水ブロックを通過した距離の合計 |
| 水中に空洞     | 表現しにくい            | 自然にできる（水ブロックを抜くだけ） |
| 水面の波       | 法線マップで表現        | ブロック上面に適用                   |

水ブロック方式は「洞窟の中に水がある」「水の中に空気ポケットがある」といった複雑な形状を自然に扱える。

### 水ブロックのレイマーチ実装

レイが水ブロックを通過するたびに距離を蓄積し、最終色にまとめて減衰をかける:

```glsl
float waterDist = 0.0;

// レイマーチ中
if (currentBlock == WATER) {
    waterDist += stepSize;
}

// 最終色に適用
vec3 absorption = vec3(0.2, 0.04, 0.03);
color = color * exp(-absorption * waterDist)
      + vec3(0.05, 0.12, 0.15) * (1.0 - exp(-absorption * waterDist));
```

### Minecraft の実装

- **バニラ**: カメラが水ブロック内にいるか判定 → フォグの色を青に、距離を短くするだけ（物理的正確さなし）
- **シェーダーMOD**（Iris/OptiFine + SEUS等）: レイに沿って水の通過距離を計算し、物理ベースの吸収・散乱を適用

### RTコアとの相性

水ブロックはRTコアと相性が良い:

- 水にはポリゴン（ブロックの面）が**ある** → RTコアで交差判定できる
- レイは水中でも**直線のまま**（屈折は水面境界で1回だけ）
- RTコアで「水の入口・出口」を高速に求め、通過距離を算出 → 減衰計算はシェーダーで行う

ブラックホールの重力レンズとの違い: BH はレイが曲がるので次の交差点が事前に分からずRTコアに渡せないが、水はレイが直線なのでRTコアの「直線レイ vs ポリゴン」交差判定がそのまま使える。

## コースティクス（水面の光の模様）

水面の波がレンズのように光を集光・分散させ、海底に網目状の明暗パターンができる現象。

### 実装手法

#### 1. テクスチャ投影（ゲームで一般的）

事前に作ったコースティクステクスチャ、またはノイズ関数を海底に投影する:

```glsl
vec2 uv1 = worldPos.xz * 0.08 + time * 0.02;
vec2 uv2 = worldPos.xz * 0.05 - time * 0.015;
float caustics = voronoi(uv1) * voronoi(uv2);  // 2層重ねるだけ
color += caustics * lightIntensity;
```

#### 2. 水面の法線から屈折計算（中程度の精度）

水面の波の法線でスネルの法則を適用し、光の集中度を計算する。

#### 3. フォトンマッピング（最も正確・映画向け）

光源から光子を飛ばし、水面で屈折させ、海底での光子密度を記録する。RTコアはこの光子追跡に有効。

|              | テクスチャ投影 | 法線ベース計算   | フォトンマッピング |
| ------------ | -------------- | ---------------- | ------------------ |
| 品質         | それっぽい     | かなりリアル     | 物理的に正確       |
| コスト       | 極めて軽い     | 中程度           | 非常に重い         |
| 動的な波対応 | 疑似的         | 対応             | 対応               |
| RTコア       | 不要           | 不要             | 有効               |
| 用途         | ゲーム         | リアルタイムデモ | 映画・オフライン   |

### Minecraft シェーダーMOD のコースティクス実装

SEUS、Continuum 等の影MODはほぼすべて**手法1（テクスチャ投影）のバリエーション**。物理的な屈折計算は一切していない。

リアルに見える理由:

| トリック                     | 効果                                               |
| ---------------------------- | -------------------------------------------------- |
| ボロノイノイズの重ね合わせ   | コースティクス特有の網目模様に見える               |
| 異なる速度・方向でスクロール | 波が干渉しているように見える                       |
| 水深に応じてぼかす           | 深い場所でコースティクスが拡散する現実の挙動を模倣 |
| 光の方向に沿って投影         | 太陽の角度でパターンがずれる                       |
| HDR + ブルーム               | 明るい部分が光って見える                           |

人間の目は「網目状の光のパターンが揺れている」だけでコースティクスと認識するため、物理的正確さは必要ない。

### 影MODが真面目にやっている部分・やっていない部分

| 効果                       | 実装                                                  | 真面目度                         |
| -------------------------- | ----------------------------------------------------- | -------------------------------- |
| コースティクス             | ノイズ関数の重ね合わせ                                | ズル                             |
| グローバルイルミネーション | スクリーンスペースのボクセルトレーシング（SEUS PTGI） | 半分真面目（画面外は計算しない） |
| 影                         | シャドウマップ (2048〜4096px) + PCF/PCSS              | 真面目                           |
| 反射                       | スクリーンスペース反射 (SSR)                          | 半分真面目（画面外は映らない）   |
| 大気散乱                   | レイリー散乱・ミー散乱を実際に計算                    | 真面目                           |
| 水面の波                   | 法線マップの多層合成                                  | それっぽい                       |

SEUS PTGI の「パストレーシング」も名前に反して本物のパストレーシングではなく、スクリーンスペースのボクセルトレーシング。画面に映っている情報だけで間接光を近似しており、カメラの後ろにある光源からの間接光は計算されない。

リアルタイムレンダリングの本質は「人間が気づかない部分を省略する」ことであり、物理的正確さよりもコスト対効果の最大化が重要。
